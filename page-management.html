Page Management

Page Creation, Move, Copy & Delete

AEM provides a seamless interface for managing pages within the Sites console. Authors and admins can create new pages, reorganize content structures, and remove unnecessary pages.

  • Create: In the Sites console, select a location and click Create ‚Üí Page. Choose a template and fill in page details like title and name.
  • Move: Select a page and choose Move to relocate it. AEM updates all internal references automatically.
  • Copy: Use the Copy and Paste options to duplicate a page, including its child pages and content.
  • Delete: Select the page and choose Delete. AEM will ask for confirmation, especially if references exist.

These operations are permission-based, ensuring only authorized users can modify site structure.

Versioning

AEM supports version control for all pages. This helps teams track changes, restore older versions, and maintain content integrity over time.

  • Creating a Version: In the page editor, click Page Info ‚Üí Create Version. Add a version label and comment for tracking.
  • Viewing Versions: Go to the Timeline and filter for version entries. You can see all previously saved versions.
  • Restoring a Version: Choose a version and click Revert to restore that content snapshot.

Versioning is crucial for audit trails, content rollback, and QA validations before publishing changes.

Timeline

The Timeline view in the Sites console provides a chronological record of changes for a selected page or folder. It's especially helpful for tracking actions over time.

  • What It Shows: Version creation, publishing, workflow events, comments, and page locks.
  • Access: Select a page and click the clock icon (Timeline) in the toolbar.
  • Commenting: Team members can add comments to the timeline for collaborative documentation.

This tool enables transparency in content management and helps resolve issues by reviewing recent activity.

Page Locking and Unlocking

Page locking ensures content integrity by preventing multiple authors from editing the same page simultaneously.

  • Automatic Locking: A page is locked automatically when an author opens it in the editor.
  • Manual Locking: Can be set from the Sites console to prevent unauthorized edits.
  • Unlocking: Only the locking user or an admin can unlock a page. This is done via the Page Info dropdown.

Locking is vital in collaborative environments to avoid overwriting changes or introducing merge conflicts.

Reference Panel

The References panel in AEM shows all relationships and dependencies associated with a page. It helps authors understand how content is reused or linked.

  • Accessing: In the Sites console, select a page and click the References icon in the top bar.
  • Includes: Language copies, live copies (MSM), experience fragments, content fragments, and referenced assets/components.
  • Use Cases: Prevent deleting referenced content, manage translations, and identify reusable assets.

This panel is essential for maintaining site integrity and avoiding accidental disruption of connected content.

Vanity URLs in AEM

A Vanity URL in AEM is a user-friendly, customizable URL path that maps to an internal content path. It allows marketers and content authors to create cleaner, shorter, and more memorable URLs for pages without changing the actual structure in the JCR.

Normally, a page in AEM might be accessed via a long path like: /content/mysite/en/products/product-a.html
With a vanity URL, you can access it via a cleaner path like: /product-a

You can assign a vanity URL through:

  • CRXDE: Open the page node (e.g., /content/mysite/en/products/product-a) and set the property: vanityPath = /product-a (type: String or String[])
  • Page Properties in Touch UI: Navigate to the page ‚Üí Click on Page Properties ‚Üí Go to the Advanced tab ‚Üí Enter the vanity URL.

üîÅ Multiple Vanity URLs

You can set multiple vanity paths as an array by entering multiple values in CRXDE or Touch UI. Each will point to the same page.

‚ùå Things to Keep in Mind

  • Vanity URLs must be unique across the repository. If two pages define the same vanity path, AEM won't know which to resolve.
  • AEM doesn't auto-redirect from the full path to the vanity URL ‚Äî both are accessible unless you configure redirection separately.
  • Vanity URLs are only resolved via the Resource Resolver and need Dispatcher to be configured properly to allow them.

üß† How AEM Resolves Vanity URLs

The org.apache.sling.jcr.resource.internal.JcrResourceResolverFactoryImpl service handles vanity URL resolution. Its key configuration properties:

  • Enable Vanity Path: Whether to allow vanity resolution at all
  • Vanity Path Bloom Filter: Used for performance optimization to cache available vanity paths
  • Maximum Entries: How many entries to allow in the cache

üö´ Disabling Vanity Paths for Performance

In large repositories, you might disable vanity path resolution or limit it to improve performance using:

  • Apache Sling Resource Resolver Factory ‚Üí Enable Vanity Path = false
  • Use Dispatcher to explicitly deny vanity paths unless required

‚úÖ Best Practices

  • Keep vanity URLs short, meaningful, and consistent with your site's brand
  • Regularly audit vanity paths to avoid conflicts or outdated mappings
  • Avoid using vanity URLs for internal-only pages or system pages

üîç Troubleshooting: Pages Not Visible in Author

  • Check Permissions: The logged-in user might not have 'jcr:read' access on the page path. Use useradmin or the CRXDE to validate ACLs.
  • Check Page Filters: The Sites Console may have filters (e.g., by name, template, modified date). Clear filters and try again.
  • Check Blueprint/Rollout Config: If using MSM (Multi-Site Manager), rollout config issues may prevent visibility.
  • Check for Deleted or Deactivated Pages: Pages that have been deactivated (unpublished) might still exist in Author but may be hidden due to filter settings.
  • Check if Page was Moved or Renamed: Use CRXDE to look for the page under its expected path. You might be searching in the wrong location.
  • Replication Status: Ensure pages weren't replicated without proper approval (check Replication tab in Page Properties).
  • Check for Workspace Corruption: Rarely, inconsistencies in JCR may cause indexing issues. Restart or reindex if needed.

Tip: Use CRXDE Lite (/crx/de) to search for the page path directly and validate its node structure.
