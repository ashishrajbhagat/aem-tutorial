Console & UI Customization

üî∞ Introduction

AEM provides out-of-the-box administrative consoles for managing content, assets, and metadata. However, real-world business needs often require these consoles to be customized for better usability, branding, or data capture.

Console Customization in AEM refers to extending or modifying the UI of tools like the:

  • DAM Properties Console
  • Folder Metadata Console
  • Sites Admin UI (List/Column View)
  • Granite-based toolbars, actions, and dialogs

These customizations improve the authoring experience, ensure consistency of metadata, and allow organizations to enforce governance rules tailored to their workflow.

üß≠ Folder Metadata Schema Customization

When working with folders in AEM Assets, you can customize the Folder Metadata Console to capture specific information like Department, Owner, or Business Unit.

üìÇ Where is the Metadata Schema stored?

Folder metadata schemas are stored under: /conf/<project>/settings/dam/admin-ui-extension/

This path allows you to define project-specific metadata layouts using the Metadata Schema Editor.

‚öôÔ∏è Steps to Customize Folder Metadata

  1. Go to Tools > Assets > Metadata Schemas in the AEM Tools console.
  2. Edit the schema associated with Folder Metadata or create a new one based on your requirements.
  3. Use the drag-and-drop editor to add fields like:
    • Text Fields (e.g., Owner Name)
    • Dropdowns (e.g., Department)
    • Date Pickers, Checkboxes, etc.
  4. Add field-level constraints, such as:
    • Required fields
    • Default values
    • Character limits
  5. Save the schema.
  6. Navigate to the folder in AEM Assets where you want to apply the schema.
  7. Open the folder's properties > go to the Advanced tab.
  8. Choose the custom metadata schema from the Metadata Schema dropdown and click Save.

‚úÖ Result

Now, when a user edits the folder properties, they will see the custom fields you've added. This improves consistency, makes folders searchable by metadata, and enables automated tagging/workflows.

üóÇ Customizing Asset Properties Console (for individual DAM assets)

AEM allows you to customize the metadata fields that appear when editing an individual DAM asset (e.g., image, PDF, video). These fields are defined using metadata schemas stored at:

/conf/global/settings/dam/admin-ui-extension/

üõ† How to Customize

  1. Go to Tools > Assets > Metadata Schemas
  2. Edit an existing schema (e.g., default) or create a new one specific to your asset type (like "Product Image Schema").
  3. Use the visual metadata schema editor to:
    • Add or remove fields like Text Input, Dropdown, Date Picker
    • Mark fields as required or read-only
    • Organize fields into Tabs (e.g., Basic Info, Rights, Usage)
  4. Save the schema and associate it with a folder or asset type as needed.

üß† Advanced Enhancements

  • Custom Logic via JavaScript: You can use granite:data or custom scripts to show/hide fields conditionally.
  • Clientlibs: Extend metadata editor behavior using custom clientlibs loaded via cq:clientLibraryFolder.
  • Render Conditions: Use granite:rendercondition nodes to control the visibility of fields based on context.

These customizations let you tailor asset editing to meet your organizational metadata needs.

üß© Overlaying Admin Consoles

Sometimes, metadata schema customization is not enough. To change the layout or behavior of admin UIs like the Sites Console, Assets Console, or CRXDE Lite, you need to overlay the original nodes under /libs.

üìç Example: Customize Sites Admin Console

The default Sites Console is located at: /libs/wcm/core/content/sites/jcr:content

To customize it:

  1. Create an overlay node at: /apps/wcm/core/content/sites
  2. Copy and adjust the structure from /libs. Only override the parts you need (avoid copying full trees).
  3. Add or modify granite:data properties to influence UI layout or field defaults.
  4. Modify cq:actions to control toolbar actions (e.g., Create, Move, Delete).

üîí Best Practices

  • Always use overlays instead of modifying /libs directly.
  • Keep overlays minimal and targeted.
  • Test overlays after AEM upgrades ‚Äî since underlying paths/components may change.

üñº Column View Customization in Consoles

AEM consoles like Sites and Assets support list and column views. You can customize the columns to show additional metadata like ‚ÄúLast Modified By‚Äù or ‚ÄúOwner‚Äù.

üõ† How It Works

  • These views are built using: sling:resourceType = granite/ui/components/coral/foundation/table
  • You can overlay the console and add a columns node to define what should appear.
  • Each column is defined with its field name, title, and sortability.

üîß Notes

  • The value property maps to the JCR property on the resource.
  • You can also use granite:data for additional behaviors or custom formatting.

üß† Using Granite Render Conditions

Granite render conditions allow you to show/hide components in the UI based on conditions like user permissions, content paths, or resource types.

üéØ Use Cases

  • Hide ‚ÄúLicense‚Äù field in the metadata editor if asset is not an image
  • Show custom tabs only for authors in a specific user group
  • Disable actions for system folders or language roots

üîß How to Use

In your dialog or schema field node, add a granite:rendercondition child:

+ licenseField
  - sling:resourceType = granite/ui/components/coral/foundation/form/textfield
  - fieldLabel = "License"
  - name = "./license"

  + granite:rendercondition
    - sling:resourceType = granite/ui/components/conditional/rendercondition
    - condition = "(jcr:content/metadata/dc:format==image/jpeg)"

üß† Custom Render Conditions

You can create your own RenderCondition Java class and register it as an OSGi service with: sling:resourceType = myapp/components/renderconditions/customCheck

This gives you full flexibility to enforce business logic.

üõ† Customize Action Buttons in Consoles

AEM consoles (Sites, DAM, etc.) include toolbar actions like Move, Publish, and Delete. You can add your own custom actions to these toolbars by overlaying the console node structure and extending cq:actions.

üéØ What Can Be Customized?

  • Add buttons like Request Legal Review, Send for Translation, etc.
  • Hook into workflows or invoke custom JavaScript on click.
  • Hide or show buttons based on user permissions or context.

üîß How to Add a Button

Overlay the relevant console (e.g., DAM or Sites):

/apps/dam/gui/content/assets/jcr:content/actions
  ‚îî‚îÄ‚îÄ legalreview
      ‚îú‚îÄ‚îÄ jcr:primaryType = nt:unstructured
      ‚îú‚îÄ‚îÄ sling:resourceType = granite/ui/components/coral/foundation/button
      ‚îú‚îÄ‚îÄ text = "Request Legal Review"
      ‚îú‚îÄ‚îÄ icon = "checkCircle"
      ‚îú‚îÄ‚îÄ granite:class = "legal-review-btn"
      ‚îî‚îÄ‚îÄ data-click-handler = "myApp.legalReviewHandler"

üì¶ JavaScript Click Handler

Extend the clientlib under clientlibs/damadmin or clientlibs/sitesadmin to handle the action:

// myApp.legalReviewHandler.js
(function(document, $) {
  "use strict";
  $(document).on("click", ".legal-review-btn", function() {
    const selectedItems = Granite.author.ui.assetFinder.getSelectedItems();
    alert("Triggering legal review for: " + selectedItems.map(i => i.name).join(", "));
    // Optionally invoke workflow or servlet
  });
})(document, Granite.$);

Note: Always include granite:rendercondition if you want to limit the visibility of the button based on role, path, etc.
