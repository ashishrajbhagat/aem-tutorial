AEM Forms

What is AEM Forms?

AEM Forms is a powerful solution within Adobe Experience Manager designed for creating, managing, and processing complex, data-driven forms at scale. It is commonly used by enterprises for customer onboarding, e-governance, healthcare forms, banking workflows, and more.

Overview and Use-Cases

  • Build forms that integrate with backend data sources (databases, REST APIs, etc.)
  • Support personalized and localized forms for global users
  • Enable workflows like approvals, e-signature, and document generation
  • Use pre-fill, validation, and conditional logic in real time

Difference between AEM Sites and AEM Forms

Aspect AEM Sites AEM Forms
Primary Use Content pages, web experiences Interactive forms, data capture
Components Page layout, content, assets Form fields, validation, submit handlers
Personalization Targeted content via ContextHub Prefilled data and user-driven logic
Back-end Binding Less common Tight integration with Form Data Models (FDMs)

When to Use AEM Forms?

  • When form data must integrate with external systems (e.g., REST, SOAP, JDBC)
  • When secure workflows and digital signatures are required
  • For large-scale transactional forms like loan applications, insurance, student onboarding
  • When compliance, accessibility, and responsive design are important

Adaptive Forms

Adaptive Forms are dynamic, responsive, and data-driven forms created using AEM Forms. They adapt in real time based on user input, profile, or external data, offering a personalized and accessible form experience.

Key Features

  • Drag-and-Drop Authoring: Easily build forms using Touch UI interface with layout containers and ready-made components.
  • Responsive Design: Automatically adapts to desktop, tablet, and mobile views.
  • Dynamic Behavior: Show/hide fields, pre-fill data, and change field properties using built-in rule editor or scripts.
  • Form Fragments: Reusable parts of a form (like address block) to maintain consistency across forms.
  • Prefill Support: Autofill form fields using user profile, session data, or external APIs.
  • Localization Support: Translate form text using i18n dictionaries and configure locale-specific content.

Supported Components

  • Text input, date pickers, numeric fields
  • Radio buttons, checkboxes, dropdowns
  • Repeatable panels and form fragments
  • Digital signature and file upload fields
  • Custom widgets and third-party integrations

Use Cases

Adaptive forms are widely used for applications such as:

  • Customer registration and onboarding
  • Health insurance claim submission
  • Government e-service forms
  • University admission processes

Form Models and Data Binding

AEM Forms supports binding form fields to structured data using Form Data Models (FDMs), XML Schema Definitions (XSDs), or JSON Schemas. This allows developers to define how form inputs map to backend systems, enabling seamless data exchange.

XML/JSON Schema-Driven Forms

  • Define reusable form structure via .xsd or .json schema files
  • Automatically generate fields based on the schema
  • Use schema-driven validation and constraints

Binding Fields to Schema Elements

Each form field can be mapped to a corresponding element in the schema using the Bind Reference property in the form editor.

  • Ensures clean data mapping on submission
  • Supports complex nested objects and repeatable elements

Form Data Models (FDM)

FDMs are AEM's way of abstracting backend data sources (REST, SOAP, Database, etc.) into a unified model for data binding.

  • Create data models using the AEM FDM UI
  • Map form fields to services for read, write, submit, or fetch
  • Supports rule-driven population and dynamic behavior

Custom Data Models

  • Custom Java services or REST endpoints can be registered and exposed to FDM
  • Enables highly dynamic data flows like fetching live prices, account info, etc.

Submit Actions and Workflows

Once a form is filled out, AEM provides multiple options for handling submission. The system is extensible and allows for both built-in and custom actions.

Out-of-the-box Submit Actions

  • Email: Sends submitted data to specified email(s)
  • REST Endpoint: Posts data to external REST APIs
  • Form Data Model: Submits data to FDM-connected backend systems
  • AEM Workflow: Triggers an AEM workflow with the form data
  • Store Content: Persists submitted content into JCR

Invoking AEM Workflows

Adaptive Forms can be configured to trigger workflows on submission. This is useful for:

  • Approval chains (e.g., loan or job application)
  • Multi-level review processes
  • Document generation and archival

Workflow models can access submitted data using MetaDataMap and process it with custom steps.

Writing Custom Submit Actions

If built-in handlers aren't enough, you can write your own Java-based submit actions:

  • Implement the com.adobe.forms.common.service.FormsSubmitAction interface
  • Register with proper OSGi configuration
  • Handle business logic such as validation, data transformation, or conditional routing

Custom submit handlers give you complete control over how the data is treated post-submission.

Form Data Model (FDM)

The Form Data Model (FDM) in AEM Forms enables seamless integration between Adaptive Forms and external data sources like REST APIs, SOAP web services, OData services, and databases (JDBC). It acts as a bridge for both reading and writing data in real time, making your forms dynamic and connected.

Overview of FDM

  • Visual interface for integrating form fields with backend services
  • Supports CRUD operations
  • Enables form field prefill, dynamic population, and post-submission data storage

Supported Integrations

  • REST: Connect to RESTful APIs with GET/POST operations
  • SOAP: WSDL-based integration with enterprise systems
  • OData: Useful for Microsoft systems like Dynamics 365
  • JDBC: Directly connect to databases using SQL queries

Creating FDMs in Touch UI

You can create a new Form Data Model by navigating to:

Tools → Forms → Data Integration → Create → Form Data Model
  • Choose a data source type (REST, SOAP, etc.)
  • Define and test service operations
  • Group data models and endpoints under one model

Mapping FDM to Form Fields

Once FDM is defined, form authors can bind fields to FDM operations:

  • Read Services: Used to prefill form fields
  • Write Services: Used during form submission to post data
  • Rules: Trigger FDM calls based on user interaction or input changes

Document Services (Optional but Powerful)

AEM Forms includes powerful Document Services that let you generate, manipulate, and render PDFs on the server side. These features are crucial for regulated industries like banking, insurance, and government that need high-fidelity documents for compliance and archival.

Generate PDF (Static or Interactive)

  • Use Form Templates or Document Templates to generate PDFs from submitted data
  • PDFs can be either interactive (fillable) or flat (non-editable)
  • Integration with Document of Record (DoR) templates to create branded, ready-to-archive PDFs

Pre-populating PDFs with Form Data

Use XML or JSON data to pre-fill PDF templates before rendering:

  • Useful for auto-generating contracts, statements, or policy documents
  • Pre-filled values can come from FDM or external APIs
  • Can be rendered on the server and sent via email or download

Flatten PDFs after Submission

  • Flattening removes all interactivity, making the PDF read-only
  • Prevents tampering and ensures regulatory compliance
  • Can be done automatically as part of the submit workflow

Server-side Document Assembly

Advanced workflows can assemble multiple fragments (e.g., form sections, dynamic content) into a single PDF called the Document of Record.

  • Use XDP or Designer templates
  • Merge attachments, dynamic text, and images into final document
  • Store or send documents using workflow actions

Note: Document services typically require the AEM Forms add-on packages and may need integration with Adobe LiveCycle components or AEM Forms on JEE.

Prefill and Personalization

One of the key strengths of AEM Forms is its ability to deliver personalized and pre-filled forms based on the user context, session information, or external data sources. This leads to better user experience and reduced data entry effort.

Using ContextHub with Adaptive Forms

ContextHub is Adobe's client-side context engine that stores and delivers contextual data about the user session. You can use it to:

  • Prefill Adaptive Form fields based on logged-in user attributes
  • Modify form visibility rules based on user segments (e.g., "existing customer" vs "new customer")
  • Support personalization in multi-step forms

To enable ContextHub integration:

  • Make sure ContextHub is enabled in your site template/page
  • Create user segments and use them in rules within Adaptive Forms
  • Access ContextHub data using expression syntax like guideBridge.resolveNode("path").value

Autofill from User Profile Data

Logged-in user profile data can be fetched and prefilled into the form automatically using:

  • Profile information from /home/users or external IDP (SSO)
  • Custom Java classes (OSGi Services) that read user info and inject into fields via custom rules
  • GuideBridge API (e.g., guideBridge.setFieldValue("email", user.email))

Prefill via FDM or URL Parameters

  • FDM Binding: Use Form Data Model's Read Services to prefill data before form loads
  • URL Parameters: Fields can also be prefilled by appending query params like ?firstName=John
  • Use custom rules to parse the parameters and populate fields

Validations and Rules Editor

AEM Forms comes with a flexible validation framework powered by its intuitive Rules Editor. Validation ensures that submitted data is complete, correct, and matches expected patterns.

Out-of-the-Box Rules Editor

The visual Rules Editor allows authors to define logic like:

  • Required field enforcement (e.g., Email must not be blank)
  • Pattern matching (e.g., mobile number should match [0-9]{10})
  • Conditional visibility (e.g., show "Company Name" only if "Business" is selected)

Rules can be created using a point-and-click interface—no coding required for simple logic.

Client-side vs. Server-side Validation

  • Client-side validation: Instant feedback, reduces server load, customizable via JavaScript
  • Server-side validation: Critical for security and final validation; executed post submission
  • Both should be used together to ensure strong validation and a good UX

Writing Custom Validation Scripts

  • You can write JavaScript in the Rules Editor using guideBridge API
  • Example: if (!value.includes("@")) { return "Invalid email"; }
  • Advanced validations can be written as OSGi services and invoked from the backend during submission
  • Use FormValidator interface (Java) for reusable server-side validation logic

Best Practice: Always validate mandatory business fields server-side to avoid client-side tampering.

AEM Forms with Workflows

AEM Forms tightly integrates with AEM Workflows to allow automation of form-driven business processes such as approvals, reviews, and escalations.

Auto-Start Workflows from Form Submissions

  • In the Submit Action settings of an Adaptive Form, select “AEM Workflow” as the submission type.
  • Specify the Workflow Model to be triggered (e.g., approval-workflow).
  • You can pass form field values into the workflow using payload metadata or transient variables.
  • Form data (e.g., XML, JSON, or attachments) is stored under /content/usergenerated and can be used in workflow steps.

Adding Approver and Review Steps

  • Use the Participant Step in your workflow to assign tasks to approvers (e.g., manager, HR, legal team).
  • Customize the workflow inbox view to show necessary form data.
  • You can configure dynamic approvers using workflow process steps and logic based on submitted form values.
  • Example: Route to finance-group if amount > 10,000.

Integration with AEM Inbox and Notifications

  • Approvers receive tasks in the AEM Inbox (accessible via /aem/inbox).
  • Custom email notifications can be triggered via workflow steps using Email Notification Process.
  • Admins can view task status, reassign, or terminate workflows via the Workflow Console.

Use Case: Vacation request forms, onboarding workflows, change approval processes, etc.

AEM Forms Portal

The AEM Forms Portal provides a structured and user-friendly interface for displaying and managing published forms in a centralized manner. It is especially helpful in government, healthcare, and financial institutions with dozens of digital forms.

Forms Listing Component

  • Out-of-the-box component that displays a list of available Adaptive Forms.
  • Supports pagination, form descriptions, last modified dates, etc.
  • Configured to pull data from a specific folder or tag-based categories.

Search and Filter Published Forms

  • Users can search forms by title, tags, or description.
  • Filter forms by type, category, or department (e.g., HR, Finance, Legal).
  • Results are displayed dynamically using AJAX without reloading the page.

Integration with Sites Pages

  • You can embed Forms Portal components inside AEM Sites pages.
  • This allows authors to mix forms with web content seamlessly.
  • Useful for landing pages, department-specific portals, and intranet solutions.

Tip: You can customize the look and feel using clientlibs and overlaying the component.

Security Considerations

Ensuring that AEM Forms are secure is critical, especially when capturing sensitive data such as personal details, IDs, or financial information.

CAPTCHA Integration

  • AEM Forms supports integration with Google reCAPTCHA (v2 and v3).
  • Add the CAPTCHA component from the AEM Forms component browser.
  • Configure site key and secret key via the cloudconfig dialog or OSGi console.
  • Helps prevent bot submissions and protects public forms.

CSRF Token Handling

  • AEM includes automatic CSRF protection for Adaptive Forms.
  • The token is injected into the form at runtime and verified on submission.
  • No extra configuration is needed unless you've overridden servlet filters.
  • If custom submit actions are used, ensure CSRF filters are respected in the POST logic.

Enabling SSL for Form Submission

  • Ensure all Adaptive Form URLs are served over HTTPS.
  • Configure your Dispatcher and Web Server (Apache/IIS) with SSL certificates.
  • Redirect all http:// traffic to https:// using the Dispatcher rewrite rules.
  • Form submissions containing sensitive data (like Aadhaar, PAN, etc.) must never go over HTTP.

Role-Based Form Access

  • Restrict form access by configuring ACLs on the form path (/content/forms/...).
  • Use user groups and permissions to allow only specific users/roles to view/submit the form.
  • You can also implement visibility conditions using ContextHub or Profile attributes.
  • Examples: Show Employee Feedback Form only to "employee-group" or disable fields for guest users.

Tip: Always validate form data on the server side before processing, even if client-side validation is in place.

Headless Forms

Headless Forms allow you to consume and submit AEM Forms from external applications (e.g., React, Angular, mobile apps) using APIs, without relying on the traditional HTML-based rendering.

Submitting via APIs

  • Use AEM Forms' /submit endpoint to POST JSON data directly.
  • Endpoints follow the structure: /content/forms/af/my-form/jcr:content/guideContainer.af.submit.json
  • Headers must include CSRF token and Content-Type: application/json.
  • Responses include success/failure messages and error field mapping.

JSON Schema Forms

  • You can design forms based on a JSON Schema that defines the structure and data types.
  • The schema is uploaded via the Data Model editor or bound directly to the form fields.
  • This enables API-driven forms that can dynamically change based on schema variations.

Headless Rendering of Adaptive Forms

  • AEM provides a Headless Adaptive Form rendering framework (AEM Forms App SDK).
  • You can render and submit forms in external apps using JSON output, rendered via your preferred frontend stack.
  • It supports dynamic behavior (rules), validation, and FDM-backed prefill – all without using the AEM UI.
  • Includes support for offline rendering (Progressive Web App scenarios).

Use Case: Embedding a self-service onboarding form inside a React-based web portal using Headless AEM Forms.

Integration with Sign Services (Adobe Sign)

AEM Forms integrates seamlessly with Adobe Sign (previously EchoSign), allowing you to add secure eSignature workflows to your Adaptive Forms or Document workflows.

Adobe Sign Integration Basics

  • Configure Adobe Sign integration under Tools > Cloud Services.
  • You'll need an active Adobe Sign enterprise account with API access.
  • Authenticate with Adobe Sign using OAuth credentials and link the AEM Forms instance.

Adding eSignature to Forms

  • Use the “Adobe Sign Block” component in the Adaptive Forms editor to place e-signature fields.
  • Configure signer roles (e.g., applicant, manager) and assign fields accordingly.
  • The form can be routed to Adobe Sign after submission using submit actions or workflows.
  • Adobe Sign will handle the digital signature, document routing, and notifications.

Tracking Signed Responses

  • After signing, documents are returned to AEM and stored (optionally) in the repository or sent over email.
  • Status can be tracked in the Adobe Sign dashboard or AEM inbox (if integrated via workflow).
  • Audit trails and timestamps of all signatures are captured automatically.

Note: You can use Adobe Sign for contracts, HR forms, onboarding agreements, etc., right from within your AEM workflows.

Deployment & Versioning

Managing large, enterprise-grade forms requires proper versioning and deployment strategies, especially across environments (dev/stage/prod).

Managing Large Forms

  • Break complex forms into fragments (reusable blocks).
  • Use Form Templates and Library Components for consistency.
  • Organize forms under logical folders in /content/forms/.

Staging and Publishing Forms

  • Create, test, and preview forms on Author instances.
  • Use AEM's standard publishing workflow to push to Publish environment.
  • You can use replication agents or automated pipelines (in AEMaaCS).
  • Forms Portal auto-syncs published forms for rendering.

Rollback & Version History

  • Forms support versioning similar to pages.
  • You can use the Timeline to restore previous versions.
  • Track changes made by authors and rollback if needed.
  • Form assets can also be managed via package manager for better deployment control.

Debugging & Troubleshooting

Like any development feature, Adaptive Forms and their services may run into issues. Proper debugging techniques can save hours of frustration.

Enabling Form Logs

  • Enable debug logs in Felix Console for packages like:
    • com.adobe.forms
    • com.adobe.aemds
  • Monitor error.log and request.log while submitting forms.

Common Issues

  • Submit button doesn't work? → Check if required fields are filled or submit action is properly configured.
  • Prefill not working? → Confirm FDM binding and data availability.
  • Adobe Sign not triggering? → Verify credentials and cloud service setup.
  • Form doesn't render? → Ensure all dependencies (clientlibs, templates) are correctly deployed.

Inspecting Bindings & Model Issues

  • Use browser DevTools to inspect form field bindings.
  • Enable debugging in the Rule Editor to trace rule evaluation.
  • Check if schema fields are correctly mapped to input fields in the Adaptive Form structure.

Tip: Use Preview mode and Console logs extensively during development to validate field behavior and submission logic.
