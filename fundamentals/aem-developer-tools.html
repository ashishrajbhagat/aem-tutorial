AEM Developer Tools

CRXDE Lite & Developer Console Usage

CRXDE Lite is AEM's in-browser development environment. It allows developers to navigate the repository, inspect nodes, properties, and make small changes directly in the JCR.

Navigating CRXDE Lite

  • Access: /crx/de/index.jsp
  • Used to browse, create, modify or delete nodes and properties.
  • Allows viewing permissions, node types, mixins, and custom components.

Developer Consoles

  • System Console: /system/console - View bundles, configurations, components.
  • Web Console: Tools for thread dumps, OSGi services, memory stats, etc.
  • CRX Explorer: /crx/explorer - Alternative lightweight repository browser.

Note: While CRXDE Lite is powerful, it is not meant for heavy development. Use it for inspection or quick fixes.

Understanding the AEM Directory Structure

AEM stores content and code in a hierarchical structure in the JCR. Understanding this is key for developing clean and upgrade-safe code.

Key Folders

  • /apps: Custom code and components. Editable and project-specific.
  • /libs: Core AEM and product code. Read-only. Never modify.
  • /content: Site content (pages, assets, etc.).
  • /conf: Editable templates and policies.
  • /etc: Deprecated but still used for tools like workflows and tags (in older versions).

Best Practices

  • Never override anything in /libs. Instead, overlay into /apps.
  • Keep components modular inside /apps/your-project/components.
  • Organize templates under /apps/your-project/templates.

CRX Package Manager

The CRX Package Manager allows developers to upload, download, build, and install AEM packages (.zip files containing JCR content).

Accessing the Package Manager

  • URL: /crx/packmgr
  • Supports building code, content, or config packages for deployment.

Creating a Package

  • Click Create Package, provide name and version.
  • Define filters (e.g., /apps/your-project, /content/your-site).
  • Build and Download the package (.zip file).

Package Filters

  • Filters define which JCR paths to include/exclude in a package.
  • Use carefully to avoid including unwanted system paths.

User Permissions and Access Control

AEM supports a robust permission model using users, groups, and access control lists (ACLs).

Managing Users & Groups

  • Access via /useradmin or /security/users.html
  • Users can be assigned to groups (e.g., content-authors, administrators).

Setting Permissions

  • Permissions include Read, Write, Modify, Delete, etc.
  • Can be set in CRXDE Lite or Touch UI (/security/permissions.html).
  • Follow the principle of least privilege (grant only what's necessary).

Tip: Always test access from non-admin accounts to validate permission settings.

Touch UI vs Classic UI

AEM offers two UI frameworks — Touch UI (modern, Coral3-based) and Classic UI (legacy ExtJS-based).

Touch UI

  • Default and recommended for authoring and development.
  • Based on Coral UI and Granite UI libraries.
  • Responsive and mobile-friendly.

Classic UI

  • Deprecated, but still used in older AEM projects.
  • Non-responsive, outdated layout.

Which One to Use?

  • Use Touch UI for all modern projects.
  • Classic UI only if supporting legacy environments.
