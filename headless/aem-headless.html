AEM as Headless CMS

What is a Headless CMS?

A Headless CMS decouples the content management backend from the front-end presentation. Instead of rendering HTML on the server, it exposes content via APIs, typically in JSON format, so that any front-end application (web, mobile, IoT, SPA, etc.) can consume and render it however it likes.

Adobe Experience Manager (AEM) supports headless delivery using multiple technologies like:

  • Content Fragment APIs (REST & GraphQL)
  • Sling Model Exporter (JSON Exporter)
  • Page Model Exporter for structured content delivery

Use Cases for AEM as Headless

  • Building SPAs using React, Angular, Vue
  • Mobile apps pulling content dynamically
  • Serving content to smart TVs or kiosks
  • Sharing content across multiple platforms from a single AEM instance

AEM's hybrid capabilities also allow combining traditional authoring with headless delivery.

Content Fragments for Headless Delivery

Content Fragments are structured content objects created using fragment models. These are ideal for headless because they separate content from layout and support reusable, channel-neutral content.

Steps to Set Up Content Fragments:

  1. Create a Content Fragment Model using schema editor under /conf
  2. Create Content Fragments based on the model
  3. Expose them via GraphQL or REST APIs

They support nested references, multilingual content, versioning, and validations - making them ideal for headless delivery.

Using AEM GraphQL API

AEM exposes a native GraphQL API for content fragments that allows flexible, efficient querying of structured content. You can:

  • Query only the fields you need
  • Filter, paginate, and sort results
  • Use persisted queries for performance and security

Endpoint:

/content/graphql/global/endpoint.json

Example Query:

query {
  allProductList {
    items {
      title
      description
      price
    }
  }
}

GraphQL queries can be tested in the AEM GraphQL IDE or executed from any frontend.

Using JSON Exporter (Sling Model Exporter)

Sling Model Exporter can expose any AEM component as JSON. You annotate your Java models using @Exporter and configure them to serialize as JSON.

Sample Java Code:

@Model(adaptables = Resource.class, resourceType = "myproject/components/card", defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
@Exporter(name = "jackson", extensions = "json")
public class CardModel {
    @ValueMapValue private String title;
    @ValueMapValue private String description;
    @ValueMapValue private String image;
}

Then, accessing the component via .model.json gives JSON output:

/content/mysite/home/_jcr_content/root/container/card.model.json

Page Model Exporter

The Page Model Exporter provides a JSON representation of the entire page structure, useful for SPAs. It includes components and their content, layout structure, etc.

Used primarily with AEM SPA Editor (React/Vue).

Example:

/content/mysite/home.model.json

This output is typically consumed by frontend frameworks to render a page dynamically based on the JSON structure.

Authoring Experience for Headless

  • Content authors still use the AEM interface to create fragments, manage metadata, and organize content
  • Authors don't worry about layout — they focus on pure content
  • Editors can preview API content via custom preview setups

Headless doesn't mean losing authoring experience — AEM supports structured, reusable, and validated content models.

Hybrid CMS: Headless + Traditional

AEM is not purely headless — it supports both traditional (full page rendering) and headless (API-driven) use cases. You can:

  • Use Experience Fragments in both SPA and traditional sites
  • Use React/Vue in components inside traditional pages
  • Expose only part of the site as API, rest via HTML

This hybrid capability is a major strength of AEM compared to pure headless CMSes.

Security & API Considerations

  • Use CSRF protection and authentication headers when exposing APIs publicly
  • Restrict access using ACLs and custom filters
  • Use Persisted Queries with GraphQL for performance and control

AEM supports token-based auth and dispatcher-level caching for APIs as well.

Best Practices

  • Design reusable and scalable content fragment models
  • Use Sling Models Exporter for granular JSON control
  • Always include metadata in your models (title, tags, etc.)
  • Test GraphQL endpoints with tools like Postman or Altair
  • Leverage Hybrid delivery when needed

Conclusion

AEM's support for headless content delivery makes it a powerful and flexible platform for modern digital experiences. Whether you're building a mobile app, an SPA, or delivering omnichannel content, AEM's headless capabilities — via GraphQL, JSON Exporter, and hybrid rendering — offer robust solutions with the power of enterprise-level content authoring.
